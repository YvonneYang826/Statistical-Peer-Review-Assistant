1 Project overview

1.1 Motivation

Scientific manuscripts often contain statistical claims that are difficult to evaluate quickly during peer review. Some problems are obvious, such as confusing a p-value with the probability that the null hypothesis is true. Other problems require a broader manuscript-level review, such as whether the primary outcome is clear, whether secondary outcomes are overinterpreted, whether subgroup claims are supported by interaction tests, and whether tables contain internal inconsistencies.

This project builds a Statistical Peer Review Assistant in R. The app is designed as a first-pass statistical screening tool for academic or scientific manuscripts. It doesn’t replace a statistician, subject-matter expert, or journal reviewer. Instead, it helps identify issues that a human reviewer may want to check.

The app supports two workflows:

  1. Whole-paper review: upload a PDF or TXT file and run a section-by-section statistical review.
  2. Claim/paragraph review: paste a single claim, abstract, table excerpt, or paragraph and run a focused statistical review.

The current Shiny app intentionally separates these workflows. Earlier versions tried to extract optional claims from uploaded whole papers, but testing showed that this was confusing because whole-paper review and claim-level review serve different purposes.

1.2 What the app does

The app combines:

  • R Shiny for the interactive interface.
  • shinychat for users’ inputs and follow-up questions.
  • ellmer for calling Anthropic Claude from R.
  • RAG using a curated CSV of statistical peer-review resources.
  • A multi-agent workflow where different agents review the text from different statistical perspectives.

2.4 Project data files

The final app uses one curated RAG baseline file:

data/RAG_baseline.csv

This file is the backend knowledge base used by claim_auditor_team.R for retrieval. It should contain the full RAG fields used by the backend, including id, category, paper_type, topic, source, source_url, full_reference, misconception, principle, warning_sign, expert_question, example, issue_or_guideline, and reviewer_question. The app can also display a simplified resource table, but the safest approach is to point the display table to the same RAG_baseline.csv file or to a companion file derived from it. This avoids a mismatch where the app displays one resource list but retrieves from another.

The RAG baseline includes sources on statistical reporting, causal inference, p-values, multiplicity, prediction modeling, diagnostic reasoning, missing data, generalizability, and common misleading concepts. The purpose of the RAG file is not to make the LLM memorize specific papers. It provides reliable statistical-review principles and reviewer questions that can be retrieved when relevant to a claim or manuscript.

The backend expects an expertise/ folder containing the specialist-agent .md prompt files. The final synthesizer is not stored as a separate .md file; it is defined directly in claim_auditor_team.R. If an expertise file is missing, the backend contains fallback prompts, but the recommended version uses the external .md files so prompts can be edited without rewriting the R code.

3.2 File upload handling

The upload helper accepts .txt and .pdf. Text files are read directly. PDF files are processed with pdftools::pdf_text().

Code 
extract_text_from_upload <- function(file_info) {
  if (is.null(file_info)) {
    stop("No file uploaded.")
  }

  ext <- tolower(tools::file_ext(file_info$name))

  if (ext == "txt") {
    text <- readr::read_file(file_info$datapath)

    if (nchar(text) < 20) {
      stop("The TXT file appears to contain too little text.")
    }

    return(text)
  }

  if (ext == "pdf") {
    if (!requireNamespace("pdftools", quietly = TRUE)) {
      stop("Package 'pdftools' is required for PDF upload.")
    }

    pages <- tryCatch(
      suppressWarnings(pdftools::pdf_text(file_info$datapath)),
      error = function(e) {
        stop(
          "PDF text extraction failed. This PDF may be encrypted, corrupted, scanned/image-based, or malformed. ",
          "Try converting it to TXT first. Original error: ", e$message
        )
      }
    )

    text <- paste(pages, collapse = "\n\n")
    text <- stringr::str_replace_all(text, "\r\n|\r", "\n")
    text <- stringr::str_replace_all(text, "[ \t]+", " ")
    text <- stringr::str_replace_all(text, "\n{3,}", "\n\n")
    text <- stringr::str_trim(text)

    if (nchar(text) < 100) {
      stop("The PDF text could not be extracted well.")
    }

    return(text)
  }

  stop("Unsupported file type. Please upload a PDF or TXT file.")
}

4.1 Knowledge-base loading

The RAG knowledge base is loaded from a CSV. The backend checks that required columns exist.

Code 
load_knowledge_base <- function(
  path = "data/RAG_baseline.csv"
) {
  if (!file.exists(path)) {
    alt_path <- file.path(find_project_root(), path)

    if (file.exists(alt_path)) {
      path <- alt_path
    } else {
      stop("Knowledge base file not found: ", path)
    }
  }

  kb <- readr::read_csv(path, show_col_types = FALSE)
  names(kb) <- stringr::str_trim(names(kb))

  required_cols <- c(
    "id", "category", "paper_type", "topic", "source",
    "source_url", "full_reference", "misconception",
    "principle", "warning_sign", "expert_question",
    "example", "issue_or_guideline", "reviewer_question"
  )

  missing_cols <- setdiff(required_cols, names(kb))

  if (length(missing_cols) > 0) {
    stop("Knowledge base is missing required column(s): ",
         paste(missing_cols, collapse = ", "))
  }

  kb <- kb |>
    dplyr::mutate(
      dplyr::across(dplyr::everything(), ~ ifelse(is.na(.x), "", as.character(.x)))
    )

  assign("kb", kb, envir = KB_WORKSPACE)
  invisible(TRUE)
}

In the final app, the backend retrieval file is:

data/RAG_baseline.csv

RAG_baseline.csv is treated as the authoritative RAG dataset and recommend deriving any displayed resource table from the same file.

The purpose of the RAG baseline is not to make the LLM memorize specific papers. It gives the agents reliable statistical-review principles, such as: report effect sizes and confidence intervals, avoid interpreting p-values dichotomously, justify causal adjustment sets, check for multiplicity, and distinguish relative from absolute risk.

4.2 Retrieval function

The retrieval function scores each knowledge-base entry using keyword overlap plus topic-specific bonuses.

Code 
retrieve_context <- function(
  claim,
  top_n = 8,
  paper_type = NULL,
  review_mode = NULL
) {
  kb <- get_knowledge_base()

  query_text <- paste(claim, paper_type %||% "", review_mode %||% "", collapse = " ")
  claim_lower <- stringr::str_to_lower(query_text)
  claim_words <- clean_words(query_text)
  claim_words <- claim_words[nchar(claim_words) >= 3]

  scored <- kb |>
    dplyr::rowwise() |>
    dplyr::mutate(
      combined_text = paste(
        category, paper_type, topic, source, source_url,
        full_reference, misconception, issue_or_guideline,
        principle, warning_sign, expert_question,
        reviewer_question, example,
        sep = "\n"
      ),
      row_words = list(clean_words(combined_text)),
      keyword_score = sum(claim_words %in% row_words),
      phrase_bonus = dplyr::case_when(
        stringr::str_detect(claim_lower, "p-value|significant|null hypothesis") ~ 8,
        stringr::str_detect(claim_lower, "causal|confounding|dag|target trial") ~ 9,
        stringr::str_detect(claim_lower, "prediction|auc|accuracy|calibration") ~ 9,
        TRUE ~ 0
      ),
      score = keyword_score + phrase_bonus
    ) |>
    dplyr::ungroup() |>
    dplyr::arrange(dplyr::desc(score), id)

  scored |>
    dplyr::slice_head(n = top_n)
}

The actual project contains more detailed phrase bonuses for p-values, multiplicity, prediction models, causal inference, randomized trials, behavioral studies, odds ratios, diagnostic testing, missing data, generalizability, Simpson’s paradox, regression to the mean, meta-analysis, and table inconsistency.

4.3 Context formatting

Retrieved rows are converted into text that can be passed into the agents.

Code 
format_context_for_agent <- function(context_rows) {
  if (is.null(context_rows) || nrow(context_rows) == 0) {
    return("No relevant knowledge base context was retrieved.")
  }

  safe_get <- function(df, row_i, col_name) {
    if (!col_name %in% names(df)) return("")
    value <- df[[col_name]][row_i]
    if (length(value) == 0 || is.na(value) || is.null(value)) return("")
    as.character(value)
  }

  context_text <- purrr::map_chr(seq_len(nrow(context_rows)), function(i) {
    paste0(
      "Knowledge Base Entry ", safe_get(context_rows, i, "id"), "\n",
      "Category: ", safe_get(context_rows, i, "category"), "\n",
      "Topic: ", safe_get(context_rows, i, "topic"), "\n",
      "Source: ", safe_get(context_rows, i, "source"), "\n",
      "URL: ", safe_get(context_rows, i, "source_url"), "\n",
      "Principle: ", safe_get(context_rows, i, "principle"), "\n",
      "Reviewer question: ", safe_get(context_rows, i, "reviewer_question"), "\n"
    )
  })

  paste(context_text, collapse = "\n---\n")
}

5 Multi-agent design

5.1 Agent list

The app uses a small specialist team rather than one large all-purpose prompt. Each specialist returns a focused intermediate output, and the final synthesizer combines those outputs into the user-facing review.

Agent Source Purpose
Claim Extractor expertise/claim_extractor.md Extracts population, exposure/intervention, outcome, comparison group, effect measure, time frame, design, denominator, and missing information.
Statistical Misconception Agent expertise/statistical_misconception.md Checks common statistical interpretation problems such as p-value misuse, missing uncertainty, denominator mismatch, odds/risk confusion, and small-sample overinterpretation.
Causal Inference Agent expertise/causal_inference.md Checks whether causal language is supported by the design and whether confounding, reverse causality, selection bias, or temporality are relevant.
Skeptical Reviewer Agent expertise/skeptical_reviewer.md Reduces overconfidence and identifies unsupported assumptions or facts that cannot be verified from the provided text.
Guardrail Coach expertise/guardrail_coach.md Converts statistical concerns into cautious, professional, user-facing language and reminds users that the app is not a substitute for expert review.
Manuscript Type Classifier expertise/manuscript_type_classifier.md Classifies the manuscript type, such as randomized trial, observational study, diagnostic accuracy study, prediction model, or unclear.
Reporting Guideline Agent expertise/reporting_guideline.md Checks whether key reporting elements appear to be present for the selected manuscript type.
Prediction Model Review Agent expertise/prediction_model_review.md Reviews prediction-model and machine-learning claims for validation, calibration, overfitting, leakage, and clinical usefulness.
Final Synthesizer defined directly in claim_auditor_team.R Combines the specialist outputs into the final structured review. This is not stored as a separate .md prompt file.

5.3 Claim-level final prompt

The claim-level synthesizer is required to return a concise structure:

# Claim Review Summary

1. Verdict
2. Main problem
3. Why this matters
4. Safer wording
5. What to check
6. Technical note

Allowed verdicts are:

Supported
Questionable
Misleading
Incorrect
Cannot assess from provided text

A key prompt revision came from Test Set 4. The app initially over-criticized good RCT abstracts because it treated normal abstract-level omissions as actual statistical flaws. The revised abstract guardrail says:

If the provided text appears to be an abstract, summary, or short excerpt, do not treat missing full-paper details as confirmed flaws.

For good randomized-trial abstracts that report design, sample size, primary endpoint, effect estimate, confidence interval, and p-value, default to Supported or Questionable, not Misleading, unless the abstract itself makes an unsupported overclaim.

Missing details from an abstract should be phrased as verify in the full paper, not not reported or prevents assessment, unless the abstract makes a strong claim that depends on the missing detail.

This significantly improved the Test Set 4 results.

6 Whole-paper workflow

6.1 Paper mapping

Whole-paper review begins with a Paper Mapper step. The purpose of this step is not to critique the paper yet. Instead, it creates a structured overview of the uploaded manuscript so that later reviewer agents can understand the study design, sample, outcomes, statistical methods, and main claims before producing a critique.

This matters because a full manuscript is usually too long and complex to review as one undifferentiated text block. The Paper Mapper turns the uploaded article into a more organized representation of the paper. Later agents can then use this map to focus on different review tasks, such as checking statistical methods, causal language, reporting transparency, internal consistency, and potential overclaims.

When a user uploads a PDF or TXT file, the app first extracts the manuscript text. The backend then searches for common manuscript section headings, including:

Abstract
Introduction
Methods
Results
Discussion
Tables / Figures
References

The function split_paper_into_sections() uses these section markers to divide the manuscript into smaller parts. If section headings cannot be detected reliably, the app does not stop the workflow. Instead, it keeps the document as a single full-text block and continues the review.

Code 
split_paper_into_sections <- function(text) {
  text <- paste(as.character(text), collapse = "\n")
  text <- stringr::str_replace_all(text, "\r\n|\r", "\n")
  text <- stringr::str_replace_all(text, "[ \t]+", " ")
  text <- stringr::str_replace_all(text, "\n{3,}", "\n\n")

  lower <- stringr::str_to_lower(text)

  find_pos <- function(patterns) {
    positions <- purrr::map_dbl(patterns, function(pat) {
      loc <- stringr::str_locate(
        lower,
        stringr::regex(pat, ignore_case = TRUE)
      )[, 1]

      if (is.na(loc)) {
        return(Inf)
      }

      as.numeric(loc)
    })

    pos <- suppressWarnings(min(positions, na.rm = TRUE))

    if (is.infinite(pos) || is.na(pos)) {
      return(NA_real_)
    }

    pos
  }

  markers <- tibble::tibble(
    section = c(
      "Abstract",
      "Introduction",
      "Methods",
      "Results",
      "Discussion",
      "Tables and Figures",
      "References"
    ),
    pos = as.numeric(c(
      find_pos(c("\\babstract\\b")),
      find_pos(c("\\bintroduction\\b")),
      find_pos(c("\\bresearch methods and procedures\\b",
                 "\\bmethods\\b",
                 "\\bmethod\\b")),
      find_pos(c("\\bresults\\b")),
      find_pos(c("\\bdiscussion\\b")),
      find_pos(c("\\btable 1\\b",
                 "\\bfigure 1\\b",
                 "\\btable 2\\b")),
      find_pos(c("\\breferences\\b"))
    ))
  ) |>
    dplyr::filter(!is.na(pos), is.finite(pos)) |>
    dplyr::arrange(pos) |>
    dplyr::distinct(section, .keep_all = TRUE)

  if (nrow(markers) == 0) {
    return(list(
      Full_Text = text
    ))
  }

  sections <- list()

  for (i in seq_len(nrow(markers))) {
    start <- markers$pos[i]
    end <- if (i < nrow(markers)) markers$pos[i + 1] - 1 else nchar(text)

    section_text <- substr(text, start, end)
    section_text <- stringr::str_trim(section_text)

    if (nchar(section_text) > 50) {
      sections[[markers$section[i]]] <- section_text
    }
  }

  sections
}

One important implementation detail is the use of purrr::map_dbl() in find_pos(). Earlier in development, I encountered an error when missing section headings produced Inf values and the code attempted to coerce them to integers. Using map_dbl() avoids that problem because the section positions may temporarily include Inf before missing values are filtered out. This made the section-splitting logic more stable across different papers and PDF extraction formats.

6.3 False-positive guardrail for whole-paper review

Testing with good randomized trials showed that the app initially overreacted to routine reviewer checks. A whole-paper false-positive guardrail was added:

If the paper appears to be a randomized, controlled, blinded, peer-reviewed clinical trial with a clear primary outcome, sample size, effect estimates, and confidence intervals, do not assign High concern unless there is a clearly confirmed statistical flaw that directly threatens the primary conclusion.

For well-reported randomized trials with a statistically supported primary outcome, default to Low concern or Low-to-Moderate concern.

Do not label secondary outcome, subgroup, or exploratory-analysis concerns as Major confirmed issue unless they directly invalidate the primary conclusion.

This guardrail improved performance on good-paper controls.

8 Evaluation and robustness testing

The evaluation was developed iteratively. The goal was not only to check whether the app can identify problems, but also whether it avoids over-criticizing strong statistical work.

8.1 Scoring system

Simple score:

  • 0 = Missed the expected issue or produced a misleading false positive.
  • 1 = Partially identified the issue or partially over-criticized the paper.
  • 2 = Correctly identified the issue or appropriately avoided false-positive severity.

A detailed 10-point score was also used internally. The 10-point score considers verdict/severity accuracy, issue detection, explanation quality, reviewer usefulness, and user-friendliness.

This scoring system was developed manually for this project. It is not a validated psychometric instrument or an official peer-review metric. The purpose was to make the evaluation process consistent across repeated tests: each app output was compared with an expected statistical-review behavior, then scored based on how well the app identified the issue and calibrated the severity of its response.

NOTE: The scoring system is developed by the creator of the app; therefore, the system is subjective, biased and could be inaccurate.

  2. RAG grounding. The app retrieves statistical-review principles from a curated knowledge base rather than relying only on the model’s internal memory.
  3. Specialist agent trace. The app separates extraction, statistical review, causal review, reporting review, skepticism, and communication guardrails.
  4. User interface. The Shiny app gives users one place to upload papers, paste claims, inspect the agent process, review RAG context, and view robustness results.
  5. Evaluation-driven prompt improvement. The false-positive guardrails were added because testing showed specific failure modes.

The most important lesson from Test Set 5 is that the app should not be presented as simply “better than ChatGPT.” In several cases, especially good RCT abstracts, the baseline LLM produced answers that were as good as or slightly better calibrated than the app. This was useful rather than discouraging: it showed that the app’s main contribution is not raw reasoning ability alone. The contribution is the surrounding workflow: fixed prompts, a curated RAG knowledge base, explicit agent roles, visible agent trace, separated review modes, and documented robustness testing.

A more accurate claim is:

The app turns LLM-based statistical peer review into a structured, documented, reproducible workflow with RAG grounding, agent specialization, and built-in robustness testing.

9 Limitations

The app has important limitations:

  1. It cannot verify raw data. It can flag table inconsistencies or missing information, but it cannot confirm fraud, fabrication, or hidden data errors without external data.
  2. It depends on PDF extraction quality. Poorly extracted text can lead to missed sections or incorrect summaries.
  3. It does not replace expert review. Clinical, economic, psychological, or biomedical conclusions require domain expertise.
  4. It may over-focus on statistical reporting. A paper can be statistically well reported but still scientifically weak, or vice versa.
  5. RAG retrieval is keyword-based. The retrieval function is transparent and simple, but it is not as semantically powerful as embedding-based retrieval.
  6. The knowledge base needs maintenance. URLs, papers, and guidelines may change over time.
  7. LLM outputs vary. Even with guardrails, repeated runs may differ slightly.

10 Potential Future Works

Future work could include:

  1. Embedding-based RAG. Use vector embeddings instead of keyword/phrase scoring.
  2. Structured JSON outputs. Force agents to return structured fields for easier evaluation.
  3. Citation validation. Automatically verify that retrieved URLs are valid.
  4. PDF table extraction. Extract tables more reliably and compare numbers programmatically.
  5. Benchmark context field. Allow users to provide known critique or replication context when testing known papers.
  6. Batch evaluation mode. Run all test claims and papers automatically and output an evaluation CSV.
  7. Deployment with persistent logs. Store anonymized evaluation logs if privacy allows.
  8. More paper types. Add dedicated modes for systematic reviews, diagnostic accuracy, causal inference papers, and prediction models.
  9. User-adjustable severity. Let users choose conservative vs. reviewer-style vs. author-friendly feedback.
  10. Video demonstration. Record a short YouTube video showing both workflows and the Evaluation tab.

# Project overview

## Motivation

Scientific manuscripts often contain statistical claims that are difficult to evaluate quickly during peer review. Some problems are obvious, such as confusing a p-value with the probability that the null hypothesis is true. Other problems require a broader manuscript-level review, such as whether the primary outcome is clear, whether secondary outcomes are overinterpreted, whether subgroup claims are supported by interaction tests, and whether tables contain internal inconsistencies.

This project builds a Statistical Peer Review Assistant in R. The app is designed as a first-pass statistical screening tool for academic or scientific manuscripts. It doesn't replace a statistician, subject-matter expert, or journal reviewer. Instead, it helps identify issues that a human reviewer may want to check.

The app supports two workflows:

1.  **Whole-paper review**: upload a PDF or TXT file and run a section-by-section statistical review.
2.  **Claim/paragraph review**: paste a single claim, abstract, table excerpt, or paragraph and run a focused statistical review.

The current Shiny app intentionally separates these workflows. Earlier versions tried to extract optional claims from uploaded whole papers, but testing showed that this was confusing because whole-paper review and claim-level review serve different purposes.

## What the app does

The app combines:

-   **R Shiny** for the interactive interface.
-   **shinychat** for users' inputs and follow-up questions.
-   **ellmer** for calling Anthropic Claude from R.
-   **RAG** using a curated CSV of statistical peer-review resources.
-   **A multi-agent workflow** where different agents review the text from different statistical perspectives.

The structure of the app is shown below:

-   Test Set 2: known-problem or stress-test papers.
-   Test Set 3: good statistical-analysis full-paper controls.
-   Test Set 4: abstract-only review.

## File upload handling

The upload helper accepts `.txt` and `.pdf`. Text files are read directly. PDF files are processed with `pdftools::pdf_text()`.

```{r upload-helper, eval=FALSE}
extract_text_from_upload <- function(file_info) {
  if (is.null(file_info)) {
    stop("No file uploaded.")
  }

  ext <- tolower(tools::file_ext(file_info$name))

  if (ext == "txt") {
    text <- readr::read_file(file_info$datapath)

    if (nchar(text) < 20) {
      stop("The TXT file appears to contain too little text.")
    }

    return(text)
  }

  if (ext == "pdf") {
    if (!requireNamespace("pdftools", quietly = TRUE)) {
      stop("Package 'pdftools' is required for PDF upload.")
    }

    pages <- tryCatch(
      suppressWarnings(pdftools::pdf_text(file_info$datapath)),
      error = function(e) {
        stop(
          "PDF text extraction failed. This PDF may be encrypted, corrupted, scanned/image-based, or malformed. ",
          "Try converting it to TXT first. Original error: ", e$message
        )
      }
    )

    text <- paste(pages, collapse = "\n\n")
    text <- stringr::str_replace_all(text, "\r\n|\r", "\n")
    text <- stringr::str_replace_all(text, "[ \t]+", " ")
    text <- stringr::str_replace_all(text, "\n{3,}", "\n\n")
    text <- stringr::str_trim(text)

    if (nchar(text) < 100) {
      stop("The PDF text could not be extracted well.")
    }

    return(text)
  }

  stop("Unsupported file type. Please upload a PDF or TXT file.")
}
```

## Retrieval function

The retrieval function scores each knowledge-base entry using keyword overlap plus topic-specific bonuses.

```{r retrieval, eval=FALSE}
retrieve_context <- function(
  claim,
  top_n = 8,
  paper_type = NULL,
  review_mode = NULL
) {
  kb <- get_knowledge_base()

  query_text <- paste(claim, paper_type %||% "", review_mode %||% "", collapse = " ")
  claim_lower <- stringr::str_to_lower(query_text)
  claim_words <- clean_words(query_text)
  claim_words <- claim_words[nchar(claim_words) >= 3]

  scored <- kb |>
    dplyr::rowwise() |>
    dplyr::mutate(
      combined_text = paste(
        category, paper_type, topic, source, source_url,
        full_reference, misconception, issue_or_guideline,
        principle, warning_sign, expert_question,
        reviewer_question, example,
        sep = "\n"
      ),
      row_words = list(clean_words(combined_text)),
      keyword_score = sum(claim_words %in% row_words),
      phrase_bonus = dplyr::case_when(
        stringr::str_detect(claim_lower, "p-value|significant|null hypothesis") ~ 8,
        stringr::str_detect(claim_lower, "causal|confounding|dag|target trial") ~ 9,
        stringr::str_detect(claim_lower, "prediction|auc|accuracy|calibration") ~ 9,
        TRUE ~ 0
      ),
      score = keyword_score + phrase_bonus
    ) |>
    dplyr::ungroup() |>
    dplyr::arrange(dplyr::desc(score), id)

  scored |>
    dplyr::slice_head(n = top_n)
}
```

# Whole-paper workflow

## Paper mapping

Whole-paper review begins with a **Paper Mapper** step. The purpose of this step is not to critique the paper yet. Instead, it creates a structured overview of the uploaded manuscript so that later reviewer agents can understand the study design, sample, outcomes, statistical methods, and main claims before producing a critique.

This matters because a full manuscript is usually too long and complex to review as one undifferentiated text block. The Paper Mapper turns the uploaded article into a more organized representation of the paper. Later agents can then use this map to focus on different review tasks, such as checking statistical methods, causal language, reporting transparency, internal consistency, and potential overclaims.
  
When a user uploads a PDF or TXT file, the app first extracts the manuscript text. The backend then searches for common manuscript section headings, including:

```text
Abstract
Introduction
Methods
Results
Discussion
Tables / Figures
References
```

The function `split_paper_into_sections()` uses these section markers to divide the manuscript into smaller parts. If section headings cannot be detected reliably, the app does not stop the workflow. Instead, it keeps the document as a single full-text block and continues the review.

```{r section-splitter, eval=FALSE}
split_paper_into_sections <- function(text) {
  text <- paste(as.character(text), collapse = "\n")
  text <- stringr::str_replace_all(text, "\r\n|\r", "\n")
  text <- stringr::str_replace_all(text, "[ \t]+", " ")
  text <- stringr::str_replace_all(text, "\n{3,}", "\n\n")

  lower <- stringr::str_to_lower(text)

  find_pos <- function(patterns) {
    positions <- purrr::map_dbl(patterns, function(pat) {
      loc <- stringr::str_locate(
        lower,
        stringr::regex(pat, ignore_case = TRUE)
      )[, 1]

      if (is.na(loc)) {
        return(Inf)
      }

      as.numeric(loc)
    })

    pos <- suppressWarnings(min(positions, na.rm = TRUE))

    if (is.infinite(pos) || is.na(pos)) {
      return(NA_real_)
    }

    pos
  }

  markers <- tibble::tibble(
    section = c(
      "Abstract",
      "Introduction",
      "Methods",
      "Results",
      "Discussion",
      "Tables and Figures",
      "References"
    ),
    pos = as.numeric(c(
      find_pos(c("\\babstract\\b")),
      find_pos(c("\\bintroduction\\b")),
      find_pos(c("\\bresearch methods and procedures\\b",
                 "\\bmethods\\b",
                 "\\bmethod\\b")),
      find_pos(c("\\bresults\\b")),
      find_pos(c("\\bdiscussion\\b")),
      find_pos(c("\\btable 1\\b",
                 "\\bfigure 1\\b",
                 "\\btable 2\\b")),
      find_pos(c("\\breferences\\b"))
    ))
  ) |>
    dplyr::filter(!is.na(pos), is.finite(pos)) |>
    dplyr::arrange(pos) |>
    dplyr::distinct(section, .keep_all = TRUE)

  if (nrow(markers) == 0) {
    return(list(
      Full_Text = text
    ))
  }

  sections <- list()

  for (i in seq_len(nrow(markers))) {
    start <- markers$pos[i]
    end <- if (i < nrow(markers)) markers$pos[i + 1] - 1 else nchar(text)

    section_text <- substr(text, start, end)
    section_text <- stringr::str_trim(section_text)

    if (nchar(section_text) > 50) {
      sections[[markers$section[i]]] <- section_text
    }
  }

  sections
}
```

One important implementation detail is the use of `purrr::map_dbl()` in `find_pos()`. Earlier in development, I encountered an error when missing section headings produced `Inf` values and the code attempted to coerce them to integers. Using `map_dbl()` avoids that problem because the section positions may temporarily include `Inf` before missing values are filtered out. This made the section-splitting logic more stable across different papers and PDF extraction formats.

After section splitting, the function `make_paper_map_text()` creates shortened excerpts from each section. These excerpts are passed to the LLM with a fixed prompt asking it to produce a structured **Paper Map**.

```{r paper-map-text, eval=FALSE}
}
```

## False-positive guardrail for whole-paper review

Testing with good randomized trials showed that the app initially overreacted to routine reviewer checks. A whole-paper false-positive guardrail was added:

``` text

Do not label secondary outcome, subgroup, or exploratory-analysis concerns as Major confirmed issue unless they directly invalidate the primary conclusion.
```

This guardrail improved performance on good-paper controls.

# Evaluation and robustness testing

The evaluation was developed iteratively. The goal was not only to check whether the app can identify problems, but also whether it avoids over-criticizing strong statistical work.

## Scoring system

**Simple score:**

-   **0** = Missed the expected issue or produced a misleading false positive.
-   **1** = Partially identified the issue or partially over-criticized the paper.
-   **2** = Correctly identified the issue or appropriately avoided false-positive severity.

A detailed 10-point score was also used internally. The 10-point score considers verdict/severity accuracy, issue detection, explanation quality, reviewer usefulness, and user-friendliness.

This scoring system was developed manually for this project. It is not a validated psychometric instrument or an official peer-review metric. The purpose was to make the evaluation process consistent across repeated tests: each app output was compared with an expected statistical-review behavior, then scored based on how well the app identified the issue and calibrated the severity of its response.

**NOTE:** The scoring system is developed by the creator of the app; therefore, the system is subjective, biased and could be inaccurate.

## Test Set 3: Good full-paper false-positive controls

**Purpose:**

Check specificity: can the app avoid falsely labeling strong statistical work as seriously flawed?


### Why these papers were considered good statistical-analysis controls

The good-paper controls were chosen to test false positives. They were not chosen because they are perfect or because an external article officially declared them “good.” Instead, they have features that a statistical reviewer should usually respect: randomized or cluster-randomized design, clear primary endpoints, sample sizes reported, effect estimates with confidence intervals, pre-specified or protocol-driven analyses, and appropriate caution around secondary or subgroup findings.

These papers are therefore useful as *specificity controls*. The expected behavior is not “find no limitations.” A good review should still ask about component outcomes, secondary endpoint multiplicity, subgroup interpretation, safety, missing data, and generalizability. The key is that those should be framed as standard reviewer checks or interpretation cautions, not as major confirmed flaws.

| Paper | Why it is a useful good-paper control |
|------------------------------------|------------------------------------|
| Postpartum hemorrhage RCT | Large cluster-randomized trial with a clearly defined composite primary outcome, reported risk ratio, confidence interval, and implementation outcomes. Useful for testing whether the app handles cluster RCTs without over-criticizing missing full-paper details. |
| Sotatercept PAH trial | Phase 3 randomized, double-blind, placebo-controlled trial with a clear primary endpoint and reported treatment difference. Useful for testing whether the app separates a supported primary endpoint from cautious interpretation of secondary endpoints and long-term outcomes. |
| Empagliflozin heart failure trial | Large randomized, double-blind clinical trial with time-to-event analysis, hazard ratio, confidence interval, and composite endpoint. Useful for testing whether the app supports the primary composite result while asking for component-specific interpretation. |
| Polatuzumab DLBCL trial | Phase 3 oncology RCT with a supported progression-free-survival endpoint but no overall survival benefit at reported follow-up. Useful for testing whether the app can support the primary endpoint while cautioning about clinical magnitude and endpoint discordance. |

The baseline LLM comparison later in the tutorial also supported using these as false-positive controls: the baseline LLM generally rated the good full-paper controls as Low concern and supported their primary statistical conclusions while still noting limitations.

Papers included:

| Paper | Expected behavior | Score |
|----------------------|----------------------|----------------------------:|
| Randomized Trial of Early Detection and Treatment of Postpartum Hemorrhage | Support strong cluster-RCT primary outcome; raise only standard checks | 2 / 2 |
| Phase 3 Trial of Sotatercept for Pulmonary Arterial Hypertension | Support primary endpoint; treat secondary/unblinding/generalizability points as cautious checks | 2 / 2 |

**Main lesson:**

The whole-paper false-positive guardrail worked. The app generally gave Low-to-Moderate concern, supported the primary endpoint, and avoided treating routine reviewer checks as major confirmed flaws.

## Test Set 5: Baseline existing-LLM comparison

After testing the app directly, I added a baseline comparison against an existing general-purpose LLM. The goal was to check whether the custom app add value beyond asking a general LLM to act as a statistical peer reviewer.

For this comparison, I used standardized prompts for three input types:

-   Single claim review,
-   Whole paper review,
-   Abstract only review.

The baseline LLM didn't use the app's Shiny interface, RAG retrieval table, specialist agents, paper mapper, or evaluation-tuned guardrails. It only received the article text or claim plus the baseline prompt. This makes the comparison useful because it separates the value of the app architecture from the value of a general LLM alone.

### Baseline prompts

To make the baseline comparison reproducible, I used standardized prompts. These were written manually for the evaluation. The baseline LLM did not receive the app's RAG table, agent trace, paper mapper, or Shiny interface. It only received one of the prompts below plus the article text, abstract, or claim.

#### Prompt 1: Single-claim baseline prompt

``` text
You are acting as a statistical peer reviewer.

I will provide one statistical or scientific claim. Your task is to evaluate whether the claim is statistically well supported based only on the text I provide.

Use this exact structure:

1. Verdict
Choose exactly one:
- Supported
- Questionable
- Misleading
- Incorrect
- Cannot assess from provided text

2. Main problem
Explain the single most important statistical issue in 2–3 sentences.

3. Why this matters
List 2–3 short bullets.

4. Safer wording
Rewrite the claim more carefully in one sentence.

5. What to check
List the most important 3 things a reader or reviewer should verify.

6. Technical note
Optional. Include only if needed. Keep this under 60 words.

Important rules:
- Use only the claim I provide.
- Do not search the internet.
- Do not use outside facts about the topic.
- Do not invent p-values, confidence intervals, sample sizes, denominators, effect sizes, or study design details.
- Missing information alone does not always mean “Cannot assess.”
- If the claim contains a recognizable statistical reasoning error, choose “Misleading” or “Incorrect,” not “Cannot assess.”
- Use “Incorrect” only when the claim directly misdefines or misinterprets a statistical concept.
- Use “Misleading” when the claim may contain some truth but omits key context or leads readers to an unsupported conclusion.
- Use “Questionable” when the claim may be reasonable but important statistical details are missing.
- Use “Supported” only when the claim is cautious and the provided text gives enough evidence to support it.
- Keep the full response concise.

CLAIM:
[Paste one claim here]
```

#### Prompt 2: Full-article or long-excerpt baseline prompt

``` text
You are acting as a statistical peer reviewer.

I will provide text from a scientific article. Your task is to evaluate whether the statistical claims are well supported by the information provided.

Please review the article using the following structure:

1. Overall concern level
Choose one:
- Low concern
- Low-to-Moderate concern
- Moderate concern
- High concern
- Cannot assess from provided text

2. Study design and statistical evidence
Briefly identify the study design and summarize what statistical evidence is provided.

3. Main statistical concerns
List the top 3–5 concerns, if any. Focus on:
- whether the design supports the conclusion
- whether effect sizes and confidence intervals are reported
- whether p-values are interpreted correctly
- whether multiple testing or subgroup claims are handled carefully
- whether causal language is justified
- whether denominators, group sizes, and outcome definitions are clear
- whether missing data, attrition, or exclusions could affect interpretation
- whether the conclusion overstates the results

4. Claims needing more cautious wording
List any claims that should be softened or clarified.

5. What to check in the full paper or supplement
List the most important items that a human reviewer should verify.

6. Final assessment
Briefly state whether the article’s main conclusion appears supported, questionable, misleading, or not assessable from the provided text.

Important rules:
- Use only the article text I provide.
- Do not search the internet.
- Do not use outside knowledge about whether the article was later criticized, replicated, or retracted.
- Do not invent p-values, confidence intervals, sample sizes, effect sizes, or study details.
- If something is missing from the provided text, say it should be checked in the full paper rather than assuming it is absent.
- Be concise but specific.

ARTICLE TEXT:
[Paste the full article text or extracted PDF text here]
```

#### Prompt 3: Abstract-only baseline prompt

``` text
You are acting as a statistical peer reviewer.

I will provide only the abstract of a scientific article. Your task is to evaluate whether the statistical claims in the abstract are appropriately worded and supported by the information given in the abstract.

Please use this exact structure:

1. Verdict

- Supported
- Questionable
- Misleading
- Incorrect
- Cannot assess from provided text

2. Main issue

4. What requires caution

Important rules:
- Use only the abstract text I provide.
- Do not search the internet.
- Do not use outside knowledge about the article.

[Paste the abstract here]
```

### What the baseline comparison showed

The baseline LLM comparison changed my interpretation of the project. It showed that a carefully prompted general LLM can perform well on many statistical review tasks. However, the app still adds value in several ways:

1.  **Reproducible workflow.** The app uses fixed prompts, fixed output structures, and fixed test sets instead of ad hoc chat interactions.
2.  **RAG grounding.** The app retrieves statistical-review principles from a curated knowledge base rather than relying only on the model's internal memory.
3.  **Specialist agent trace.** The app separates extraction, statistical review, causal review, reporting review, skepticism, and communication guardrails.
4.  **User interface.** The Shiny app gives users one place to upload papers, paste claims, inspect the agent process, review RAG context, and view robustness results.
5.  **Evaluation-driven prompt improvement.** The false-positive guardrails were added because testing showed specific failure modes.

The most important lesson from Test Set 5 is that the app should not be presented as simply “better than ChatGPT.” In several cases, especially good RCT abstracts, the baseline LLM produced answers that were as good as or slightly better calibrated than the app. This was useful rather than discouraging: it showed that the app's main contribution is not raw reasoning ability alone. The contribution is the surrounding workflow: fixed prompts, a curated RAG knowledge base, explicit agent roles, visible agent trace, separated review modes, and documented robustness testing.

A more accurate claim is:

The app turns LLM-based statistical peer review into a structured, documented, reproducible workflow with RAG grounding, agent specialization, and built-in robustness testing.


# Limitations

The app has important limitations:

1.  **It cannot verify raw data.** It can flag table inconsistencies or missing information, but it cannot confirm fraud, fabrication, or hidden data errors without external data.
2.  **It depends on PDF extraction quality.** Poorly extracted text can lead to missed sections or incorrect summaries.
3.  **It does not replace expert review.** Clinical, economic, psychological, or biomedical conclusions require domain expertise.
4.  **It may over-focus on statistical reporting.** A paper can be statistically well reported but still scientifically weak, or vice versa.
5.  **RAG retrieval is keyword-based.** The retrieval function is transparent and simple, but it is not as semantically powerful as embedding-based retrieval.
6.  **The knowledge base needs maintenance.** URLs, papers, and guidelines may change over time.
7.  **LLM outputs vary.** Even with guardrails, repeated runs may differ slightly.

# Potential Future Works

Future work could include:

1.  **Embedding-based RAG.** Use vector embeddings instead of keyword/phrase scoring.
2.  **Structured JSON outputs.** Force agents to return structured fields for easier evaluation.
3.  **Citation validation.** Automatically verify that retrieved URLs are valid.
4.  **PDF table extraction.** Extract tables more reliably and compare numbers programmatically.
5.  **Benchmark context field.** Allow users to provide known critique or replication context when testing known papers.
6.  **Batch evaluation mode.** Run all test claims and papers automatically and output an evaluation CSV.
7.  **Deployment with persistent logs.** Store anonymized evaluation logs if privacy allows.
8.  **More paper types.** Add dedicated modes for systematic reviews, diagnostic accuracy, causal inference papers, and prediction models.
9.  **User-adjustable severity.** Let users choose conservative vs. reviewer-style vs. author-friendly feedback.
10. **Video demonstration.** Record a short YouTube video showing both workflows and the Evaluation tab.
